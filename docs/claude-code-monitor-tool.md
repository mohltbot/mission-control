## What Is It?

## Why It Matters for Mission Control

Our current scheduled-agent architecture (e.g., `bensbites-implementation`, autopush,
OpenClaw triage) runs on polling loops: the agent wakes up, checks a condition, and either
acts or sleeps. Every poll burns tokens even when nothing has changed.

The Monitor tool replaces the polling loop with an **event-driven callback** — Claude
only activates when a specified condition fires (file change, webhook, cron tick, etc.).

### Projected Impact

| Workflow | Current Approach | With Monitor Tool |
|---|---|---|
| bensbites-implementation | Cron fires, agent fetches + processes always | Agent wakes only when new email arrives |

| ArchTrack heartbeat | Periodic ping to check agent health | Event-driven on dashboard state change |
